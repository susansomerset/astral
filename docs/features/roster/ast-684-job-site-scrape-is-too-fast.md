### Description

## Purpose

Roster job-page discovery depends on Playwright capturing **visible job listing content** before **select_job_page** decides whether a URL is the careers list, has no openings, or needs another link. On staging batch `select_job_page-54d02162-370a-4548-b166-664bc2350bbb` (PagerDuty `careers.pagerduty.com/jobs/search`), the scraped page text contained careers chrome and filters but **not** the job titles that were visibly present in the browser. **select_job_page** then returned **JOBLIST_TITLES** with titles that do not appear in the scraped text — a false positive that would send downstream parsing down the wrong path.

Susan needs roster scrapes to **wait until dynamic job listings have rendered** (or exhaust a bounded wait) before visible text is extracted and sent to the LLM, so **select_job_page** and **parse_job_list** operate on content that actually reflects what a candidate would see.

## Functional scope

* **Readiness before extract (all paths):** For **every** roster Playwright navigation that produces visible text fed into **select_job_page** — stored **job_site** re-scrape, possible-job-link discovery, and standalone **select_job_page** dispatch — wait for job-listing content to appear, not only document **load** / brief settle. Use a shared, config-driven readiness strategy (timeouts and retry limits in config, not ad hoc per call site).
* **Bounded patience:** If listings still do not appear after the configured wait, do **not** silently proceed as if the page were complete. Record that the scrape was incomplete and surface it through the **select_job_page** outcome (below), not a false **JOBLIST_TITLES**.
* **New response type — JOBSITE_SCRAPE_ISSUE:** Extend **select_job_page** with response type **JOBSITE_SCRAPE_ISSUE** for when Grace sees the same failure mode Susan saw: careers/listing page shell (filters, headers, empty listing region) without verifiable job titles in the scraped text. Grace reports the issue in structured payload fields Susan can inspect in Execution History — clearer than a hallucinated **JOBLIST_TITLES**.
* **Terminal on scrape issue:** When **select_job_page** returns **JOBSITE_SCRAPE_ISSUE**, transition the company to a dedicated state that reflects an incomplete careers scrape (**JOBSITE_SCRAPE_ISSUE**), and **do not** invoke downstream **parse_job_list** or other further AI analysis on that run.
* **Honest LLM input:** The assembled **PAGE N** sections passed to **select_job_page** must reflect post-wait visible text. When debug is on, each scraped page logs whether listing-like content was observed, approximate size, and wait outcome — per the product debug contract ([AST-538](https://linear.app/astralcareermatch/issue/AST-538/improve-quality-of-debug-logging)): index-style headers (`index N/M`, URL, outcome), working lines prefixed `|`, long payloads truncated (first 15 / omission count / last 15).
* **Regression coverage:** Include at least one automated check that documents the expected wait/readiness behavior so a future shorten-wait change is caught before staging.

## Boundaries

* Does **not** change **select_job_page** prompt prose beyond what Susan authors in Manage Tasks to teach Grace when to emit **JOBSITE_SCRAPE_ISSUE**; Susan owns prompt content.
* Does **not** fix unrelated **find_job_page** hang / missing agent-data issues tracked separately ([AST-666](https://linear.app/astralcareermatch/issue/AST-666/find-job-page-is-well-and-truly-borked)); this ticket is premature scrape / empty listings content quality.
* Does **not** add UI debug logging (backend only per code rules).
* Does **not** replace vendor-specific **parse_job_list** DOM logic or **gazer** JD scrape paths — scope is roster job-page **discovery** scrapes that feed **select_job_page**.
* Does **not** introduce unbounded waits or per-company hardcoded selectors in product code; readiness knobs belong in config.
* Must not break companies that already scrape successfully on the current timing, **JOBS_FOUND** / **TO_WATCH** state transitions, or today’s **job_site**-aware **find_job_page** path ([AST-669](https://linear.app/astralcareermatch/issue/AST-669/find-job-page-job-doesnt-display-the-agent-content) / [AST-674](https://linear.app/astralcareermatch/issue/AST-674/job-site-aware-find-job-page-and-dispatch-agent-data-inspection-find)).

## Acceptance criteria

1. Re-running roster discovery against a JS-heavy careers listing URL equivalent to the PagerDuty repro (dynamic list populated after initial paint) captures **verbatim job titles** in the visible text passed to **select_job_page**, not only page chrome and filters.
2. With **debug=True** on the same path, logs show per-page index headers with URL, wait/readiness outcome, and whether listing-like content was found; detail lines use the `|` prefix; payloads over 50 lines are truncated per [AST-538](https://linear.app/astralcareermatch/issue/AST-538/improve-quality-of-debug-logging).
3. When listings never render within configured bounds (or scraped text is shell-only), **select_job_page** returns **JOBSITE_SCRAPE_ISSUE** — not **JOBLIST_TITLES** with fabricated titles — and the company lands in **JOBSITE_SCRAPE_ISSUE** state with **no** downstream **parse_job_list** hop on that run.
4. Existing roster flows that already returned valid **JOBLIST_TITLES** / **JOBLIST_NO_JOBS** / **TRY_LINKS** on static or fast-loading sites continue to pass without requiring longer waits on every site.
5. Susan can verify fix via Execution History on a staging **select_job_page** batch: **no_cache** page sections contain the job titles visible in the browser for the repro case, or **JOBSITE_SCRAPE_ISSUE** with a clear agent report when they do not.
